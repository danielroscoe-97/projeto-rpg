### Decisões Técnicas

**Story 3.1+3.2 — Turn Notifications**
- Abordagem de derived state puro (sem `useEffect` para mostrar/esconder)
- `isPlayerTurn` = `combatants[currentTurnIndex].is_player`
- `isPlayerUpcoming` = `nextCombatantId` matches qualquer `is_player === true` combatant
- Limitação conhecida: sem tracking de "qual player é este client", o banner "Você é o próximo" aparece para todos os player clients quando qualquer personagem de jogador é o próximo — comportamento aceitável para MVP
- Animação Framer Motion `scale: [1, 1.02, 1]` com `repeat: Infinity` evita CSS custom keyframes
- `navigator.vibrate?.([200])` — chamada opcional (nem todos os browsers suportam)

**Story 3.4 — Role Selection**
- `role` column tem `DEFAULT 'both' NOT NULL` — a condição original `!profile?.role` era sempre falsa; corrigida para `count === 0` (heurística: nenhuma campanha criada = novo usuário)
- Skip button não salva nada (já tem o default 'both') — só navega para dashboard

**Story 4.1 — GM Notes**
- `simpleMarkdown()` inline (sem react-markdown dep): escapa HTML *antes* das regex → XSS-safe
- localStorage persiste open/close state por sessionId (UX: reabre onde estava)
- `debounceRef` e `savedTimerRef` com `useRef<ReturnType<typeof setTimeout>>(undefined)` — cleanup explícito no unmount

**Story 4.2 — File Sharing**
- Magic bytes validation server-side (não confia no `Content-Type` do browser)
- Path no Storage: `{session_id}/{uuid}_{filename}` — UUID do Storage é diferente do UUID do DB record
- Bug corrigido no QA: `file_path` precisava estar na resposta do POST para o signed URL funcionar imediatamente após upload (sem precisar recarregar a página)
- `session-files` bucket precisa ser criado manualmente no Supabase Dashboard (não é criável via migration SQL)

**Story 4.4 — Invite Accept**
- RLS de `player_characters` só permite insert pelo owner da campanha (DM)
- Solução: server action `acceptInviteAction` com `createServiceClient()` (service role key) — valida o invite antes de bypass RLS
- `player_characters` não tem coluna `user_id` no schema original — personagem é scoped à campanha, não ao usuário (limitação do schema V1)
