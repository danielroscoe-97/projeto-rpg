Fluxo Pos-Combate

PocketDM — Ultima atualizacao: 05/04/2026 — Referencia tecnica do fluxo pos-combate nos 3 modos

Bugs Corrigidos (commit acd69b3)

Bug 1 — Recap sumia apos refresh do navegador

O log de combate fica na memoria do navegador. Se a pagina recarregar durante o combate (F5, tab morrendo no celular, hot reload), o log some. O sistema so checava o log pra decidir se mostrava o recap — agora tambem checa se alguem perdeu HP.

Bug 2 — Numero de rounds ia pra zero

O numero de rounds tambem vinha do log. Se o log sumia, rounds = 0. Agora usa o round real do combate como fallback.

Bug 3 — DM ficava travado se relatorio crashasse

Se o buildCombatReport desse erro, nada acontecia e o DM ficava preso. Agora pula pro feedback do DM se der erro.

5. Condicao para mostrar o Recap

A condicao e a mesma nos 3 modos (parity rule):

flowchart LR A["Log tem atividade? dano causado/recebido cura ou mortes"] -->|OU| C{"MOSTRAR RECAP"} B["Alguem perdeu HP? current_hp menor que max_hp ou is_defeated"] -->|OU| C D["Nenhum dos dois"] --> E{"PULAR RECAP"} style C fill:#2d4a1a,stroke:#4ade80,color:#fff style E fill:#4a1a1a,stroke:#f87171,color:#fff

Isso garante que o recap aparece mesmo se o navegador recarregar no meio do combate — o log some mas o HP dos combatentes sobrevive no banco de dados.
